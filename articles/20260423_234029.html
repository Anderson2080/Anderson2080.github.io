一、算力账本：伪命题与物理极限

Test-Time Compute 的鼓吹者宣称，通过增加推理时的计算量（如 Chain-of-Thought、Self-Consistency），可以提升模型能力。但这是对算力效率的彻底误读。我们以 70B 参数模型为例，在 FP8 精度下，单次 Token 生成需要约 140 TFLOPS 的矩阵乘法和 2.8 GB 的 KV Cache 读写。当上下文长度从 4K 扩展到 100K 时，KV Cache 大小从 0.8 GB 膨胀至 20 GB，而 HBM3e 的带宽上限为 3.2 TB/s。这意味着，在 100K 上下文下，单次推理的 KV Cache 读写延迟将从 250 ns 飙升至 6.25 μs——这还不包括跨节点通信的 NVLink 拥塞。更致命的是，HBM 堆叠的 TSV 穿孔热密度已达 80 W/cm²，每增加一层堆叠，良率下降 15%。三星的 12 层 HBM3e 方案，其热管理成本已占封装总成本的 40%。

二、架构解剖：为什么现有方案是纸上谈兵

硅光互联被吹捧为“打破内存墙”的终极方案。但物理参数不会说谎：当前硅光调制器的能效为 1.5 pJ/bit，而同等距离的铜互连（如 NVLink 4.0）仅为 0.5 pJ/bit。在 1m 距离内，硅光的光电转换损耗使其总能耗比铜高 3 倍。更关键的是，硅光互联的微秒级延迟（包括 SerDes 和激光器启动时间）在 Test-Time Compute 场景下是灾难性的。当 Agent 进行多轮对话时，KV Cache 的碎片化导致缓存命中率低于 30%，迫使系统频繁进行跨节点缓存迁移。每次迁移需经过硅光链路，其 2 μs 的往返延迟将使单次推理的总延迟从 100 ms 增加至 500 ms。而 SRAM 的面积局限更令人绝望：在 3nm 工艺下，1 MB SRAM 占用 0.5 mm²，而 70B 模型的 KV Cache 需要至少 20 GB 的 SRAM 才能实现零碎片化——这需要 10,000 mm² 的芯片面积，远超任何单片 SoC 的物理极限。

三、商业账本：算力泡沫的清算时刻

华尔街的估值模型建立在“算力需求每 18 个月翻倍”的假设上。但 Test-Time Compute 的物理瓶颈将打破这一曲线。我们测算，一个支持 100K 上下文、1000 QPS 的推理集群，其 HBM 总带宽需求为 3.2 PB/s，对应 1000 颗 H100 GPU。按当前 H100 售价 3 万美元计算，硬件成本为 3000 万美元，而年电力成本（按 700W TDP、0.1 美元/kWh 计算）为 61.3 万美元。但更隐蔽的成本来自 KV Cache 碎片化导致的利用率下降：实际有效吞吐量仅为理论峰值的 40%，这意味着每 Token 的推理成本从 0.01 美分飙升至 0.025 美分。当 Agent 应用（如代码生成、多轮对话）的 Token 消耗量达到训练阶段的 10 倍时，推理成本将超过训练成本。对冲基金若继续押注“推理算力”的线性增长，将在 2025 年 Q2 遭遇物理定律的清算——届时，HBM4 的 6.4 TB/s 带宽仍无法满足 1M 上下文的 KV Cache 需求，而硅光互联的微秒级延迟将使 Agent 应用的用户体验彻底崩溃。
